# MT5 Gold Robot v3.4 — финальный технический пакет

Эта версия подготовлена на базе v3.3. Она завершает только технические доработки перед длительным тестом на DEMO-счёте.

## Главное ограничение

Торговая стратегия не изменена.

Без изменений остались:

- правила входа BUY и SELL;
- система оценки сигнала;
- EMA, MACD и ATR;
- расчёт стоп-лосса;
- цели 1, 2 и 3;
- перенос стоп-лосса;
- выход по обратному сигналу;
- риск на сделку;
- лимиты Daily Guard и Account Guard;
- полный объём открытия и полного закрытия позиции.

Частичные открытия и частичные закрытия не добавлялись.

## Что добавлено

### 1. Защита от повторного ордера при неоднозначном ответе MT5

Перед отправкой заявки робот сохраняет `pending_open` с уникальными:

- `request_id`;
- комментарием ордера вида `mt5bot_1234abcd`;
- временем сигнала;
- направлением;
- объёмом;
- счётом и сервером.

Если `order_send()` не вернул результат либо вернул ответ, при котором нельзя достоверно понять итог исполнения, робот:

1. не отправляет повторный ордер;
2. сохраняет `pending_open`;
3. включает защиту состояния;
4. блокирует новые сделки;
5. проверяет позицию и историю MT5;
6. восстанавливает сделку, если исполнение найдено.

К неоднозначным относятся ответы MT5 о размещённой заявке, истечении времени ожидания и потере соединения во время отправки.

Если процесс аварийно завершился между сохранением `pending_open` и получением ответа MT5, после запуска робот также сначала проверит позицию и историю. Повторная заявка по той же свече не отправляется.

### 2. Проверка свежести котировки непосредственно перед входом

Перед реальным `order_send()` робот проверяет время последнего tick.

Настройки находятся в `config/monitoring.json`:

```json
"entry_tick_max_age_seconds": 30,
"entry_tick_max_future_skew_seconds": 5
```

Если котировка устарела, не содержит корректного времени или её время заметно находится в будущем, ордер не отправляется. Эта проверка действует только перед открытием новой позиции и не изменяет торговый сигнал.

### 3. Единая граница торгового дня Daily Guard

Торговый день определяется по часовому поясу:

```text
Europe/Kyiv
```

Одна и та же граница дня теперь используется для:

- создания нового состояния Daily Guard;
- выборки закрытых сделок из истории MT5;
- расчёта дневного результата.

Внутренние границы истории передаются в UTC с учётом летнего и зимнего времени Киева.

### 4. Ротация журнала событий

Журнал `events_YYYY_MM.log` автоматически архивируется при достижении заданного размера.

Настройки:

```json
"events_log_max_bytes": 10485760,
"events_log_backup_count": 5
```

По умолчанию:

- до 10 МБ на один текущий журнал;
- до 5 архивных файлов;
- торговые CSV не удаляются и не объединяются с журналом событий.

Также убраны некоторые повторяющиеся записи, которые могли появляться в каждом цикле без изменения состояния.

### 5. Русские информационные сообщения

Пользовательские сообщения в Telegram, консоли и основных журналах переведены на русский язык:

- направление сделки;
- причины сигнала;
- состояние MT5;
- состояние Telegram;
- защита счёта;
- дневная защита;
- защита состояния;
- ответы и ошибки торговых заявок;
- открытие, перенос стоп-лосса и закрытие сделки.

Внутренние технические коды, например `BUY`, `SELL`, `ORDER_SEND_RESULT_UNKNOWN`, номера retcode, названия полей JSON и идентификаторы MT5, сохраняются без перевода. Это необходимо для совместимости state и логики восстановления. При выводе пользователю рядом используется русское описание.

## Установка на локальный ПК

1. Остановить в PyCharm:

```text
run_bot.py
run_watchdog.py
```

2. Создать резервную копию текущей папки проекта.

3. Распаковать архив поверх:

```text
D:\Роботы форекс\gold_robot\mt5_gold_robot_bot
```

4. Сохранить свои файлы:

```text
config\accounts.json
config\telegram.json
state\XAUUSD_state.json
logs\
```

Они не входят в архив и не должны заменяться.

5. Запустить сначала:

```text
run_watchdog.py
```

6. Затем запустить:

```text
run_bot.py
```

## Что проверить локально

### Обычный запуск

В Telegram:

- защита состояния — норма;
- MT5 доступен;
- котировки доступны;
- торговля разрешена;
- получение команд Telegram работает.

### Устаревшая котировка

Этот сценарий обычно возникает сам при закрытом рынке или потере котировок. При наличии сигнала робот должен отказаться от открытия и вывести русское сообщение о том, что котировка устарела. Он не должен отправлять ордер.

### Неоднозначный ответ ордера

Искусственно вызывать такой ответ на торговом счёте не требуется. Защитная логика проверена изолированным тестом. В реальной работе при подобном ответе должны появиться `pending_open` и защитная блокировка без повторного ордера.

### Граница дня

После полуночи по Киеву Daily Guard должен создать состояние новой даты, а дневной результат должен учитывать только сделки нового киевского дня.

### Ротация логов

При обычной работе достаточно убедиться, что журнал продолжает записываться. Архивы `.1`, `.2` и далее появятся только после достижения лимита размера.

## Выполненные проверки

- `compileall` для всех Python-файлов;
- проверка отсутствия некорректных имен аргументов;
- свежая котировка допускает подготовку ордера;
- устаревшая котировка блокирует ордер;
- котировка с некорректным будущим временем блокирует ордер;
- `None`, timeout, connection и placed обрабатываются как неоднозначный результат;
- неоднозначный результат не считается отказом и не разрешает слепой повтор;
- границы дня проверены для летнего и зимнего времени Киева;
- проверен 23-часовой день перехода на летнее время;
- проверена ротация журнала;
- проверена уникальность `request_id` и комментария ордера;
- `strategy.py`, `indicators.py`, `structure.py` и `strategy_settings.json` совпадают с v3.3 байт-в-байт.

Фактическое открытие и закрытие через Windows MetaTrader 5 в среде сборки не выполнялось. Перед переносом на VPS версия должна поработать на локальном DEMO-счёте.
